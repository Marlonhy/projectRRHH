# Guía Técnica y de Despliegue (RRHH Permissions App)

Este documento detalla la arquitectura del sistema y los pasos necesarios para desplegar la aplicación en un servidor interno de la compañía.

## 🏗️ Stack Tecnológico

La aplicación utiliza un stack moderno y eficiente diseñado para ejecutarse localmente o en red interna sin dependencias externas pesadas.

### 1. Frontend (Interfaz de Usuario)
- **Framework**: [React](https://reactjs.org/) (con TypeScript).
- **Herramienta de Construcción**: [Vite](https://vitejs.dev/) (proporciona velocidad extrema en desarrollo y builds ligeros).
- **Estilos**: Vanilla CSS con variables personalizadas (Modern UI Design).
- **Comunicación**: Axios para peticiones REST al backend.

### 2. Backend (Servidor de API)
- **Entorno**: [Node.js](https://nodejs.org/).
- **Framework**: [Express.js](https://expressjs.com/).
- **Lenguaje**: TypeScript (compilado a JS para ejecución).
- **Autenticación**: JWT (JSON Web Tokens) para sesiones seguras y Bicrypt para hashing de contraseñas.
- **Almacenamiento Periférico**: Librería Multer para almacenar estáticamente PDFs e imágenes en `/backend/uploads`.

### 3. Base de Datos
- **Motor**: [SQLite](https://sqlite.org/).
- **Persistencia**: Se guarda en un archivo local (`rrhh_permisos.db`) dentro de la carpeta `backend/data/`.
- **Ventaja**: No requiere instalar un servidor SQL por separado (MySQL/PostgreSQL), lo que facilita el respaldo simplemente copiando el archivo y la carpeta `/uploads/`.
- **Control de Acceso**: La tabla `usuarios` incluye un campo `activo` para gestionar bajas de personal sin borrar datos.

## 🚀 Guía de Despliegue en Servidor Interno

Para poner la aplicación en producción en un servidor de la compañía (Windows o Linux), sigue estos pasos:

### Requisitos Previos
- Instalar [Node.js (LTS)](https://nodejs.org/en/download/).
- Las terminales deben tener acceso a la red interna.

### Paso 1: Preparar el Backend
1. Entra a la carpeta `backend`.
2. Crea un archivo `.env` basado en el `.env.example`. Asegúrate de configurar:
   - `PORT=3000`
   - `JWT_SECRET=una_clave_muy_segura`
   - Configuración de correo (SMTP).
3. Instala las dependencias: `npm install`.
4. Compila el código: `npm run build` (si existe el script) o usa `ts-node` para ejecución directa con `npm start`.
5. Asegurarse de que el directorio `backend/uploads` también esté listado y expuesto al servidor HTTP estáticamente.

### Paso 2: Preparar el Frontend
1. Entra a la carpeta `frontend`.
2. Edita `src/services/api.ts` para que la `BASE_URL` apunte a la IP del servidor (ej: `http://192.168.1.50:3000/api`).
3. Instala dependencias: `npm install`.
4. Genera el paquete de producción: `npm run build`.
5. Esto creará una carpeta `dist`.

### Paso 3: Servir la Aplicación
Existen dos formas recomendadas para mantenerla activa 24/7:

#### Opción A: Usando PM2 (Recomendado)
PM2 es un gestor de procesos que reinicia la app si el servidor se apaga.
```bash
npm install -g pm2
# En backend
pm2 start dist/index.js --name rrhh-backend
# En frontend (puedes servirlo con un servidor estático)
npm install -g serve
pm2 start serve -- -s dist -l 80 --name rrhh-frontend
```

#### Opción B: Usando IIS (Windows Server)
1. Instala el módulo "HttpPlatformHandler" o usa un "Reverse Proxy" hacia el puerto 3000 de Node.
2. Mapea la carpeta `dist` del frontend a un sitio web en el puerto 80.

## 📂 Estructura de Archivos Clave
- `backend/src/controllers`: Lógica de negocio y reglas de vacaciones y festivos (`holidays.ts`).
- `backend/src/models/schema.ts`: Definición de las tablas SQLite.
- `frontend/src/pages`: Pantallas de la aplicación con paridad visual.
- `backend/data/rrhh_permisos.db`: **ARCHIVO MÁS IMPORTANTE N1** (Base de datos).
- `backend/uploads/`: Directorio donde reposan archivos **MÁS IMPORTANTE N2**.

## 🔄 Funcionamiento del Cálculo Automático
El sistema calcula los días ganados multiplicando los días transcurridos desde la fecha de ingreso por `15/360`. Cada vez que el servidor detecta que ha pasado un año, el saldo acumulativo crece naturalmente (15 días anuales), permitiendo que un colaborador guarde sus días de un año anterior para el siguiente de forma transparente. Las fechas cruzadas se evalúan usando inteligencia de festivos (ej: Ley Emiliani).
