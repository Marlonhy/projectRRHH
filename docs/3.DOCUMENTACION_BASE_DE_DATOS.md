# 🗄️ Documentación de la Base de Datos - RRHH Viajar LTDA

El sistema utiliza **SQLite** como motor de base de datos a través de la librería `better-sqlite3`. La base de datos se encuentra físicamente en `backend/data/rrhh_permisos.db`.

## 📌 Esquema de Tablas

### 1. Tabla: `Usuarios`
Es la tabla central que almacena la información de todo el personal.
- **id** (PK): Entero autoincremental.
- **email**: Único y obligatorio (se usa para el Login).
- **password**: Hash de seguridad (Bcrypt).
- **nombre** / **apellido**: Información personal.
- **documento**: Identificador nacional / documento de identidad único del empleado (nuevo registro validado en UI).
- **rol**: `colaborador`, `director`, `gerente`, `rrhh`, `admin`.
- **director_id** (FK): Referencia al `id` de otro usuario (jerarquía).
- **cargo**: Puesto del empleado.
- **fecha_ingreso**: Fecha clave para el cálculo automático de vacaciones.
- **activo**: Estado del usuario (1 para activo, 0 para inactivo). **IMPORTANTE**: No borramos usuarios para no perder historial, solo cambiamos su estado a 0. Esto bloquea inmediatamente su acceso al sistema y lo oculta de las listas activas por defecto.

### 2. Tabla: `PermisosDisponibles`
Almacena los saldos de días de cada usuario por año.
- **usuario_id** (FK): Referencia a `Usuarios`.
- **dias_disponibles**: Días de vacaciones cargados para ese año (15 base anual).
- **dias_libres**: Días libres otorgados.
- **dias_sabados**: Sábados compensatorios acumulados.
- **dias_calamidad**: Total de días tomados por calamidad (incremento acumulativo).
- **dias_incapacidad**: Total incapacidades médicas.
- **dias_cita_medica**: Total de citas médicas solicitadas.
- **dias_licencia**: Días por licencias no remuneradas tomadas.
- **ano**: El año al que corresponde el saldo.
- **Relación**: Única por `usuario_id` y `ano`.

### 3. Tabla: `SolicitudesPermisos`
Almacena todas las peticiones realizadas por los trabajadores.
- **usuario_id** (FK): Quien pide el permiso.
- **jefe_id** (FK): Quién debe aprobarlo (asignado automáticamente).
- **fecha_salida** / **fecha_regreso**: Rango de fechas.
- **tipo_permiso**: Clasificación entre los 7 tipos disponibles (vacaciones, sabado, incapacidad, etc).
- **estado**: `pendiente`, `aprobado`, `rechazado`.
- **dias_solicitados**: Cantidad de días hábiles calculados (sin domingos o festivos).
- **razon_rechazo**: Comentario opcional en caso de denegación.
- **soporte**: Identificador opcional (ruta/URL) para archivos anexados como PDFs, certificados de incapacidad, etc.
- **aprobado_gerente**: Semáforo adicional para jerarquías gerenciales (1/0).

### 4. Tabla: `HistorialAusencias`
Registros manuales creados por RRHH que no pasan por el flujo de aprobación.
- **usuario_id** (FK): Empleado afectado.
- **tipo**: Categoría de la ausencia.
- **dias_utilizados**: Días que se restan del balance de ser aplicable.
- **registrado_por** (FK): El administrador que hizo el registro.

---

## 🔗 Relaciones Recomendadas
- **Jerarquía**: `Usuarios.director_id` conecta con `Usuarios.id`. Esto permite que un Director solo vea a su equipo de Colaboradores.
- **Integridad**: Todas las tablas tienen `ON DELETE CASCADE` para que, si eliminas un usuario, se borre todo su historial automáticamente sin dejar datos huérfanos.

## 📈 Cálculo Dinámico (Lógica)
El sistema **no lee solo una columna** para mostrar los días; hace este cálculo en tiempo real ejecutado primariamente por `calculateDynamicBalance()` en la API:
`Saldo Vacaciones = (Días ganados desde ingreso calculando años flotantes) - (Suma de Solicitudes Aprobadas) - (Suma de Ausencias Manuales)`.
